# שלב ד – תכנות בסיס הנתונים (PL/pgSQL)

בשלב זה הטמענו לוגיקה עסקית מורכבת בצד השרת באמצעות שפת **PL/pgSQL**. התוכניות פותחו ונבדקו על גבי בסיס הנתונים המאוחד **IntegratedDB** שנוצר בשלב ג', תוך שימוש במבחר עשיר של אלמנטים תכנותיים (Cursors, Ref Cursors, לולאות, תנאים, טיפולי חריגות, פקודות DML מרובות, רשומות, וטריגרים).

כל התוכניות חולקו לקבצים נפרדים בתיקיית `phase4`, ובוצע גיבוי מעודכן של בסיס הנתונים בשם `backup4`.

## פונקציה 1: שליפת פרטי אטרקציה באמצעות Ref Cursor
* **שם הקובץ:** `fn_get_attraction_details_cursor.sql`
* **תיאור מילולי:** 
  הפונקציה מקבלת מזהה אטרקציה (`attraction_id`), בודקת האם היא קיימת בבסיס הנתונים (ואם לא - זורקת שגיאה מתאימה). במידה והיא קיימת, הפונקציה פותחת **Explicit Ref Cursor** (מצביע דינמי לתוצאות שאילתה) המאחד מידע על כלל הכרטיסים שהוזמנו לאטרקציה זו יחד עם הדירוגים והביקורות שניתנו לה. המצביע מוחזר לקורא (למשל לתוכנית הראשית) לצורך עיבוד וסריקה.

### קוד הפונקציה:
```sql
CREATE OR REPLACE FUNCTION fn_get_attraction_details_cursor(p_attraction_id INT)
RETURNS refcursor AS $$
DECLARE
    v_cursor_name refcursor := 'attraction_details_cursor';
    v_exists BOOLEAN;
END;
... (המשך הקוד בקובץ fn_get_attraction_details_cursor.sql)
```
> [!NOTE]
> שימוש ב-Ref Cursor מאפשר חיסכון בזיכרון בצד השרת ושליפת נתונים יעילה וממוקדת לפי דרישת הלקוח.

## פונקציה 2: חישוב הנחת נאמנות מרובת-שכבות ללקוח
* **שם הקובץ:** `fn_calculate_customer_loyalty_discount.sql`
* **תיאור מילולי:** 
  הפונקציה מחשבת דינמית את אחוז הנחת הנאמנות המגיע ללקוח על בסיס פעילותו בבסיס הנתונים.
  החישוב מתבצע לפי שלוש שכבות (Tiers):
  1. **נפח הזמנות:** מספר ההזמנות שביצע הלקוח בסטטוס Paid/Confirmed (עד 8% הנחה).
  2. **סכום השקעה כולל:** סך כל התשלומים של הלקוח במערכת (עד 7% הנחה).
  3. **מעורבות בקהילה:** שימוש ב-**Explicit Cursor** בלולאה לצורך ספירת ביקורות שכתב הלקוח ומספר התגובות (Reactions) שקיבל עליהן (עד 5% הנחה).
  * **בדיקות תקינות וחסימות:** לקוח מושעה או לקוח שיש נגדו תלונות תוכן מוצדקות (Review Reports שסווגו כ-`UPHELD`) יקבל 0% הנחה באופן מיידי. גובה ההנחה המרבי מוגבל ל-20% (Cap).

## פרוצדורה 1: ביטול פעילות אטרקציה, החזר כספי וסינון ביקורות
* **שם הקובץ:** `pr_bulk_process_refunds_and_moderation.sql`
* **תיאור מילולי:** 
  מיועדת למקרים בהם אטרקציה נסגרת ביום מסוים עקב תקלה או כוח עליון. הפרוצדורה מקבלת מזהה אטרקציה ותאריך מבוקש. 
  היא מבצעת:
  1. סריקת כל ההזמנות המושפעות בעזרת **Explicit Cursor**.
  2. עדכון ההזמנות לסטטוס `CANCELLED` והפחתת הסכום היחסי של האטרקציה מסך ההזמנה הכולל.
  3. עדכון טבלת התשלומים (`PAYMENT`) ורישום פרטי ההחזר בטבלת תיעוד החזרים (`refund_log`).
  4. ביטול/מחיקה רכה (`is_deleted = TRUE`) של כל הביקורות השליליות שנכתבו לאותה אטרקציה באותו יום ספציפי (מכיוון שהן אינן רלוונטיות עקב הסגירה).

## פרוצדורה 2: חסימת לקוחות מפירי משמעת וניקוי תוכן מדווח
* **שם הקובץ:** `pr_moderator_review_cleanup.sql`
* **תיאור מילולי:** 
  פרוצדורת ניהול ומודרציה אוטומטית. היא סורקת באמצעות **Explicit Cursor** לקוחות שפרסמו תוכן במערכת.
  עבור כל לקוח, היא סופרת את כמות הדיווחים המוצדקים (`admin_decision = 'UPHELD'`).
  אם לקוח עובר את רף הדיווחים המותר (Threshold) שהועבר כפרמטר:
  1. חשבונו מסומן כמושעה (`is_suspended = TRUE`).
  2. כל הביקורות הפעילות שכתב נמחקות מחיקה רכה ומתווסף להן תיוג `[Suspended Content]`.
  3. הליך החסימה מתועד בטבלת ביקורת מנהלים (`moderation_audit`) לצרכי מעקב.

## טריגר 1 (UPDATE): תיעוד שינויי סטטוס והחזרת מלאי כרטיסים
* **שם הקובץ:** `tr_audit_booking_status_change.sql`
* **תיאור מילולי:** 
  טריגר מסוג **AFTER UPDATE** המקושר לטבלת `BOOKING`.
  כאשר מבוצע עדכון בסטטוס ההזמנה, הטריגר בודק האם חל שינוי בערך הסטטוס.
  במידה והסטטוס החדש עודכן ל-`CANCELLED`:
  * הטריגר מריץ לולאה על כלל הכרטיסים המשויכים להזמנה ב-`BOOKINGTICKET`.
  * עבור כל כרטיס, הטריגר מחזיר את כמות הכרטיסים שהוזמנו בחזרה למלאי הזמין (`available_quantity`) בטבלת `TICKET`.
  * ללא קשר לסטטוס החדש, כל מעבר סטטוס מתועד באופן מלא בטבלת ההיסטוריה `booking_audit_log` למעקב פיננסי.

## טריגר 2 (BEFORE INSERT): חסימת פעולות של לקוחות מושעים
* **שם הקובץ:** `tr_prevent_suspended_customer_activity.sql`
* **תיאור מילולי:** 
  טריגר מסוג **BEFORE INSERT** המשמש כאמצעי הגנה קשיח במערכת.
  הטריגר מקושר לשלוש טבלאות מרכזיות בו-זמנית: `BOOKING`, `REVIEW`, ו-`REVIEWREACTION`.
  בכל ניסיון הכנסת רשומה חדשה לאחת מטבלאות אלו, הטריגר מוודא מול טבלת `CUSTOMER` שהלקוח המבצע אינו מושעה (`is_suspended = FALSE`).
  במידה והלקוח מושעה, הטריגר קוטע את העסקה (Abort Transaction) וזורק חריגה אישית מוגדרת מראש עם קוד שגיאה ייעודי (`D0001`), ובכך מונע כל פעילות זדונית מצד משתמשים חסומים.

* **שם הקובץ:** `main_program_1.sql`
* **תיאור מילולי:** 
  בלוק אנונימי (`DO $$`) המדגים הרצה משולבת של פונקציית ההנחה ופרוצדורת ההחזרים.

## תוכנית ראשית 2: הרצת תהליך מודרציה וצריכת Ref Cursor
* **שם הקובץ:** `main_program_2.sql`
* **תיאור מילולי:** 
  בלוק אנונימי מורכב המדגים שילוב של מודרציה וצריכת Ref Cursor:
  1. מזמן את הפרוצדורה `pr_moderator_review_cleanup` עם רף חסימה של דיווח אחד (Threshold = 1) כדי לבצע השעיה של משתמשים מפירי תקנון.
  2. קורא לפונקציה `fn_get_attraction_details_cursor` לקבלת Ref Cursor של אטרקציה מס' 12.
  3. מריץ לולאת `FETCH` מפורשת השולפת שורה-אחר-שורה מהמצביע, ומדפיסה בצורה מיושרת ומעוצבת את רשימת הכרטיסים והדירוגים לקונסול, תוך סגירת ה-Cursor בצורה בטוחה בסיום.
